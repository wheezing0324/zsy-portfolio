PRD / Requirement Workbench

客户需求到 PRD 工作台

模拟产品经理将多角色客户反馈转化为功能范围、流程设计、优先级和验收标准的完整过程。

3 APIs 覆盖数据加载、反馈新增、验收状态回写
背景 目标 用户 功能 Axure 接口 验收 增强版 测试部署

1. 产品背景

B 端产品经理实习岗位要求候选人能完成需求调研、需求文档撰写、原型设计支持、测试验收和反馈跟进。本作品用于模拟产品经理将客户反馈转化为结构化 PRD 的完整流程。

2. 产品目标

  • 把多角色反馈沉淀为功能范围、流程设计和验收标准。
  • 通过新增反馈、切换 PRD Tab、勾选验收项展示前后端交互。
  • 体现 B 端产品需求分析、文档结构和交付闭环能力。

3. 目标用户与核心场景

产品经理整理客户反馈并输出 PRD；客户成功和运营提交续费场景需求；测试同学根据验收标准确认功能是否达到上线要求。

核心场景是将运营负责人、客户成功、测试同学的反馈整理成可沟通、可开发、可验收的文档。

4. 功能需求

客户反馈池

  • 展示反馈角色和原始反馈内容。
  • 支持新增客户反馈，提交后刷新列表。

PRD 摘要

  • 按 Tab 展示功能范围、流程设计、验收标准。
  • 功能范围区分 P0 / P1，避免首版范围失控。

验收标准

  • 支持勾选验收项完成状态。
  • 勾选后调用后端 PATCH 接口并刷新状态。

5. Axure 原型设计

01反馈池

角色、反馈内容、新增反馈表单。

02PRD Tab

功能范围、流程设计、验收标准。

03新增反馈

提交后刷新反馈池。

04验收勾选

勾选后显示状态变化。

05状态提示

展示后端加载和写入结果。

6. 数据与接口

GET/api/prd

返回客户反馈、标签、PRD 分区和验收标准。

POST/api/prd/feedback

提交 role 和 text，新增客户反馈。

PATCH/api/prd/acceptance

提交 id 和 done，更新验收项状态。

7. 验收标准

  • 页面打开后可以从后端加载 PRD 数据。
  • 新增反馈后，反馈池新增一条记录。
  • 勾选验收项后，状态能在后端内存中保留。
  • 后端不可用时，页面仍能展示静态 fallback 内容。
  • 移动端下反馈池和 PRD 摘要上下排列。

8. 增强版功能说明

任务流

  • 查看反馈池并识别角色、场景、痛点。
  • 录入业务场景、业务价值、紧急程度、实现复杂度。
  • 系统根据评分生成 P0 / P1 / P2 优先级。
  • 同步刷新用户故事、版本计划、风险提示和 PRD 完整度。

新增字段

FIELDscenario

业务场景，如续费分析、异常识别、报表导出。

FIELDvalue / urgency / complexity

用于计算需求优先级。

RULEvalue + urgency - complexity

分数大于等于 7 为 P0，5-6 为 P1，其余为 P2。

9. 测试与部署

本地 Node 服务端口为 4173。主要测试覆盖 GET 数据加载、POST 反馈新增、PATCH 验收状态更新。部署方式为本地静态页面 + Node HTTP API，无需构建工具。
